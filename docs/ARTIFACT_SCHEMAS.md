# PlanAnvil — Artifact Schemas

> Canonical machine state is JSON. Markdown is used for human-readable contracts and narrative reports.  
> The implementation MUST ship matching JSON Schema files under `.agents/skills/plan-anvil/schemas/`.

## 1. General conventions

All canonical JSON documents MUST:

- use UTF-8 and LF line endings;
- use schema version `1.1.0` for the first implementation of contract 2.1;
- use RFC 3339 UTC timestamps;
- use SHA-256 hashes as lowercase hexadecimal prefixed with `sha256:`;
- reject unknown enum values;
- reject unspecified fields unless the schema explicitly permits them;
- end with a newline;
- be written atomically when mutable.

Identifier formats:

```text
Plan:        PG-YYYYMMDD-HHMMSS-XXXX
Run:         YYYYMMDDTHHMMSSZ_<PLAN-ID>_<slug>
Stage:       STAGE-01, STAGE-02, STAGE-03A
Requirement: REQ-03-01
Criterion:   AC-03-01
Risk:        RISK-03-01
Control:     CTRL-03-01
Checkpoint:  CHECKPOINT-03-VERIFIED
```

Deleted identifiers are never reused.

## 2. Committed versus local artifacts

Committed artifacts MUST NOT contain absolute local filesystem paths, usernames, local service locations or private machine identifiers.

The ignored local artifacts are:

```text
.pursue/SYSTEM_PROFILE.local.md
.pursue/runs/<RUN-ID>/local-state.json
```

Validators must prove that both files are ignored and absent from every commit.

## 3. `manifest.json`

Purpose: immutable committed run identity and repository-relative path map.

```json
{
  "schema_version": "1.1.0",
  "plan_id": "PG-20260712-143000-A1B2",
  "run_id": "20260712T143000Z_PG-20260712-143000-A1B2_example",
  "created_at": "2026-07-12T14:30:00Z",
  "generator_version": "0.1.0",
  "repository": {
    "fingerprint": "sha256:...",
    "base_branch": "main",
    "base_sha": "40-hex-sha",
    "planning_branch": "pursue/plan/PG-20260712-143000-A1B2/example"
  },
  "paths": {
    "run_root": ".pursue/runs/20260712T143000Z_PG-20260712-143000-A1B2_example",
    "plan": "PLAN.md",
    "state": "state.json",
    "compliance": "compliance.json",
    "traceability": "traceability.json",
    "repository_profile": ".pursue/SYSTEM_PROFILE.md",
    "local_state": "local-state.json"
  },
  "contract_versions": {
    "implementation_spec": "2.1",
    "plan_contract": "1.0.0",
    "artifact_schema": "1.1.0"
  }
}
```

Validation rules:

- `base_sha` exists in the repository;
- the planning branch descends from `base_sha`;
- the fingerprint uses stable repository identity, not a local path;
- committed paths are repository-relative and remain inside the control root;
- no field contains an absolute local path.

## 4. `local-state.json`

Purpose: ignored machine-specific locator and resume data.

```json
{
  "schema_version": "1.1.0",
  "run_id": "20260712T143000Z_PG-20260712-143000-A1B2_example",
  "updated_at": "2026-07-12T14:35:00Z",
  "paths": {
    "source_worktree": "/absolute/source/repository",
    "planning_worktree": "/absolute/planning/worktree",
    "local_profile": "/absolute/planning/worktree/.pursue/SYSTEM_PROFILE.local.md"
  },
  "hashes": {
    "local_profile": "sha256:..."
  }
}
```

Rules:

- the file is ignored and never committed or pushed;
- path values are absolute and normalized for the host OS;
- the run ID matches `manifest.json`;
- resume fails safely when required paths are missing or the profile hash differs;
- the final terminal output may display these paths, but committed reports may not copy them.

## 5. `state.json`

Purpose: current generator or later execution state in the durable control root.

```json
{
  "schema_version": "1.1.0",
  "revision": 7,
  "updated_at": "2026-07-12T14:42:00Z",
  "mode": "PLAN_GENERATION",
  "status": "ARTIFACTS_GENERATED",
  "current_stage": null,
  "current_phase": "DETERMINISTIC_VALIDATION",
  "next_action": {
    "type": "RUN_VALIDATOR",
    "target": "PLAN.md"
  },
  "last_checkpoint": null,
  "open_blockers": [],
  "artifact_hashes": {
    "PLAN.md": "sha256:..."
  }
}
```

Allowed modes:

```text
PLAN_GENERATION
PLAN_EXECUTION
```

Generator states:

```text
NEW
SOURCE_PREFLIGHT_PASSED
GIT_READY
PLANNING_WORKTREE_READY
PROFILE_READY
INSTRUCTION_MAP_READY
ANALYSIS_READY
ARTIFACTS_GENERATED
DETERMINISTICALLY_VALID
BLIND_REVIEW_WRITTEN
COMPARISON_VALID
PLAN_COMMITTED
STOPPED
BLOCKED
FAILED
```

Execution states are generated by the plan contract and include at least:

```text
EXECUTION_READY
EXECUTION_IN_PROGRESS
READY_FOR_LOCAL_INTEGRATION_TEST
AWAITING_USER_VALIDATION
USER_ACCEPTED
USER_REJECTED
BLOCKED_BY_UNRESOLVED_FAILURE
```

Rules:

- `revision` increments by exactly one per accepted update;
- `next_action` contains exactly one action;
- terminal states use `next_action.type = "NONE"`;
- artifact hashes match before transition;
- stale revision updates are rejected;
- execution updates may change control state but not silently change the approved plan.

## 6. `compliance.json`

Purpose: current run-specific compatibility and enforcement record.

```json
{
  "schema_version": "1.1.0",
  "verified_at": "2026-07-12T14:35:00Z",
  "codex": {
    "version": "unknown",
    "model": "unknown",
    "permission_mode": "unknown",
    "project_trust": "UNKNOWN",
    "hook_mode": "HOOKS_UNAVAILABLE"
  },
  "capabilities": [
    {
      "id": "CAP-SKILL-DISCOVERY",
      "status": "DOCUMENTED",
      "evidence": []
    }
  ],
  "unsupported": [],
  "warnings": []
}
```

Allowed capability statuses:

```text
VERIFIED
DOCUMENTED
FAILED
BLOCKED
UNKNOWN
NOT_APPLICABLE
```

A `FAILED`, `BLOCKED` or `UNKNOWN` capability required by active behavior blocks `PLAN_READY`.

The file records current requirements and current run results only. It is not an archive.

## 7. `traceability.json`

```json
{
  "schema_version": "1.1.0",
  "requirements": [
    {
      "id": "REQ-01-01",
      "text": "The endpoint rejects unauthenticated requests.",
      "stages": ["STAGE-01"],
      "criteria": ["AC-01-01"]
    }
  ],
  "criteria": [
    {
      "id": "AC-01-01",
      "stage": "STAGE-01",
      "risks": ["RISK-01-01"],
      "controls": ["CTRL-01-01"],
      "evidence_type": "AUTOMATED_TEST"
    }
  ],
  "gaps": []
}
```

Gap types:

```text
UNCOVERED_REQUIREMENT
STAGE_WITHOUT_REQUIREMENT
CRITERION_WITHOUT_EVIDENCE
RISK_WITHOUT_CONTROL
CONTROL_WITHOUT_BEHAVIOR
PUBLIC_BEHAVIOR_WITHOUT_REQUIRED_APPROVAL
```

Any critical gap blocks readiness.

## 8. Risk files

Each `risks/RISK-*.json` contains:

```json
{
  "schema_version": "1.1.0",
  "id": "RISK-03-01",
  "stage": "STAGE-03",
  "level": "HIGH",
  "description": "Migration may leave old and new records inconsistent.",
  "source": "Repository analysis",
  "affected_components": ["database", "worker"],
  "probability": "MEDIUM",
  "impact": "HIGH",
  "detection": ["CTRL-03-02"],
  "criteria": ["AC-03-01"],
  "controls": ["CTRL-03-01", "CTRL-03-02"],
  "mitigation": "Use a resumable expand-and-migrate flow.",
  "rollback": "Restore the verified recovery point before switching.",
  "status": "OPEN"
}
```

Risk levels:

```text
LOW
MEDIUM
HIGH
```

Risk statuses:

```text
OPEN
MITIGATED
ACCEPTED_BY_USER
CLOSED
REALIZED
```

Only the user may accept an otherwise unmitigated high risk when policy permits acceptance.

## 9. Checkpoints

Checkpoint files are immutable after creation.

```json
{
  "schema_version": "1.1.0",
  "id": "CHECKPOINT-03-VERIFIED",
  "created_at": "2026-07-12T17:00:00Z",
  "mode": "PLAN_EXECUTION",
  "stage": "STAGE-03",
  "phase": "VERIFICATION",
  "result": "PASS",
  "git": {
    "branch": "pursue/PG-.../example",
    "head": "40-hex-sha",
    "worktree_role": "TASK",
    "status": "CLEAN"
  },
  "tests": {
    "summary": "42 passed, 0 failed",
    "evidence": ["tests/results/STAGE-03.json"]
  },
  "risks": {
    "open": [],
    "closed": ["RISK-03-01"]
  },
  "agent_tree": {
    "status": "COMPLIANT",
    "evidence": ["reports/validation/agent-tree-STAGE-03.json"]
  },
  "write_scope": {
    "status": "COMPLIANT",
    "evidence": ["diffs/STAGE-03.json"]
  },
  "next_action": {
    "type": "BEGIN_STAGE",
    "target": "STAGE-04"
  },
  "recovery": [
    "Read manifest.json",
    "Read local-state.json",
    "Read state.json",
    "Verify this checkpoint hash",
    "Reconcile Git state"
  ],
  "artifact_hashes": {}
}
```

Checkpoint results:

```text
PASS
FAIL
BLOCKED
```

A checkpoint is valid only while Git and artifact evidence matches. Absolute worktree paths are resolved from ignored local state, not stored in the committed checkpoint.

## 10. Generation and execution locks

Canonical paths:

```text
run-root/.generation-lock
run-root/.execution-lock
```

Each lock uses exclusive-create semantics and contains:

```json
{
  "schema_version": "1.1.0",
  "run_id": "...",
  "owner": {
    "hostname_hash": "sha256:...",
    "pid": 1234,
    "session_id_hash": "sha256:..."
  },
  "created_at": "2026-07-12T15:00:00Z",
  "heartbeat_at": "2026-07-12T15:02:00Z",
  "command": "plan-anvil"
}
```

Locks are local runtime files and SHOULD be ignored. If lock metadata is committed for audit, direct host and session identifiers must be hashed as above.

A lock cannot be removed automatically merely because its timestamp is old. Owner liveness or session evidence must also show inactivity.

## 11. Reports

Narrative reports are Markdown with a JSON metadata sidecar.

```text
reports/plan-review/blind-review.md
reports/plan-review/blind-review.json
```

Example sidecar:

```json
{
  "schema_version": "1.1.0",
  "report_type": "BLIND_PLAN_REVIEW",
  "created_at": "2026-07-12T14:50:00Z",
  "author_role": "plan-anvil-reviewer",
  "inputs": {
    "PLAN.md": "sha256:..."
  },
  "result": "FAIL",
  "findings": [
    {
      "id": "FINDING-01",
      "severity": "HIGH",
      "summary": "Rollback is undefined for the migration stage."
    }
  ],
  "markdown_hash": "sha256:..."
}
```

Comparison verifies `markdown_hash` and never rewrites blind-review files.

Committed reports must not include local absolute paths.

## 12. Stage briefs

Stage briefs remain Markdown and begin with deterministic metadata:

```yaml
---
schema_version: "1.1.0"
stage_id: "STAGE-03"
outcome: "Migrate existing records without downtime."
classification: "STATEFUL"
requirements: ["REQ-03-01"]
criteria: ["AC-03-01", "AC-03-02"]
risks: ["RISK-03-01"]
dependencies: ["STAGE-02"]
applicable_instructions:
  - path: "AGENTS.md"
    sha256: "..."
allowed_write_paths:
  - "src/migration/**"
  - "tests/migration/**"
---
```

The implementation may use a restricted parser for this exact grammar or a JSON sidecar. Core scripts must not add a general YAML runtime dependency.

## 13. Atomic update protocol

For every mutable canonical JSON file:

1. read and schema-validate the current file;
2. verify the expected revision or hash;
3. construct the complete replacement document;
4. write a sibling temporary file;
5. flush and fsync where supported;
6. replace atomically;
7. fsync the containing directory where supported;
8. reread and validate;
9. retain immutable checkpoints and evidence.

On Windows, fail explicitly when file locking prevents safe replacement.

## 14. Path safety

Before any write:

- resolve the candidate path;
- reject traversal and symlink escapes;
- compare normalized paths correctly on case-insensitive filesystems;
- reject submodule writes unless explicitly in scope;
- reject `.git` paths except through dedicated Git commands;
- record the approved repository-relative glob and resolved target.

Allowed-path matching uses repository-relative POSIX-style paths.

## 15. Schema evolution

Schema versions use semantic versioning.

- patch: clarification that does not change accepted documents;
- minor: backward-compatible optional fields or a compatible new artifact;
- major: incompatible field or semantic change.

A run keeps the schema version with which it was created. Migration is explicit and never automatic across a major version.
